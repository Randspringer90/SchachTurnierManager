# Demo script and English voice-over

Target duration: 2:50–2:55. Keep the final public video under three minutes. Do not show private
paths, Wi-Fi credentials, signing material or real player data.

## 0:00–0:20 — problem and audience

**Picture:** Title, quiet club-tournament setup, then the installed app.

**Voice-over:**

> Volunteer chess organisers need registrations, pairings, results and reports to work together.
> SchachTurnierManager turns one Windows PC into a local-first tournament workstation for clubs and
> small opens—without an account, advertising or a cloud requirement.

## 0:20–0:45 — installation and clear start

**Picture:** Start the already installed candidate; show Overview and the two first-start actions.

**Voice-over:**

> The per-user Windows setup is designed for a double-click installation. The interface starts with
> five clear areas. A director can create a tournament or explicitly open a synthetic demo; no
> sample data is inserted automatically.

## 0:45–1:20 — demo, participants and FIDE Dutch

**Picture:** Open the demo, Participants, then Round; briefly show FIDE Dutch in settings/audit and
pairing preview.

**Voice-over:**

> Our demo contains eight fictional players and one completed round. Optimal V2 stays the default
> pairing method, while FIDE Dutch is an opt-in choice with a relevant initial-colour setting and an
> auditable preview. We do not claim FIDE certification; the goal is transparent, testable pairing
> behaviour.

## 1:20–1:50 — Android companion

**Picture:** Galaxy S25 and PC on the same Wi-Fi; connect to a private address, open one board,
submit a synthetic result, show confirmation on both devices.

**Voice-over:**

> The Android companion connects directly to the tournament PC on the same private Wi-Fi. It checks
> the local service identity and lets a helper submit a result. There is no cloud relay. The desktop
> asks for a clear confirmation and keeps an undo path.

If the real-device test has not passed, omit this segment and say the companion is a candidate
awaiting device validation. Never present a mock recording as device evidence.

## 1:50–2:15 — standings and compatibility

**Picture:** Updated Standings; reveal extra tie-breaks; More → exports; show generated filenames,
not private paths.

**Voice-over:**

> Standings update from the recorded results. Common columns stay readable, with detailed tie-breaks
> available on demand. Tournament directors can exchange Swiss-Manager CSV and TRF16 data, print
> round sheets and create local backups.

## 2:15–2:40 — local-first and safety

**Picture:** Simple local PC ↔ phone diagram; audit view; repository tests without tokens/paths.

**Voice-over:**

> Tournament state remains in a local SQLite database. Core operation has no tracking, ads or cloud
> dependency. Pairings and manual interventions remain auditable, and required Android artifacts
> are accepted only by exact path, hash, type and provenance—not by a blanket binary bypass.

## 2:40–2:55 — Codex and GPT-5.6

**Picture:** Public-safe Build Week changelog, tests and this primary Codex collaboration report.

**Voice-over:**

> This project existed before Build Week. During the submission period, Codex and GPT-5.6 helped
> implement and review the documented additions: pairing and format compatibility, safer mobile
> integration, progressive UX, synthetic demo data, tests and submission evidence. Owner decisions
> and earlier human or other-tool contributions remain credited separately.
